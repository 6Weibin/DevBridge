# 技术预研：AI 历史对话本地文件持久化

## 1. 预研结论

- 结论：可行。
- 推荐方案：后端独立历史存储模块，采用“会话目录 + 追加写 JSONL 分段 + 冷分段 GZIP 压缩 + 轻量元数据索引 + 分页读取”。
- 不使用数据库，不依赖浏览器 `localStorage`，不在 React 状态中加载全部历史。
- 存储能力按“单会话至少 1M Token、会话数量至少 1000 个”设计，代码层面不设置 1000 个硬上限。
- 是否需要 PoC：建议。正式开发前用 1000 会话和单会话 1M Token 数据验证写入延迟、分页加载、压缩率、内存峰值和异常恢复。
- 下一步：确认 1M Token 的业务含义后进入 spec/design，再实施迁移。

## 2. 预研目标

### 2.1 新需求目标

1. 历史聊天从浏览器 `localStorage` 迁移到本机文件系统。
2. 保持轻量化，不引入关系型数据库、嵌入式数据库或独立数据服务。
3. 至少支持 1000 个历史会话，并能快速展示、分页加载、重命名和删除。
4. 单会话可以完整保存至少 1M Token 的消息及必要工具执行记录，不因前端存储限制截断正文。
5. 控制磁盘占用、写放大、CPU、IO 和内存占用，兼顾 Electron 与普通浏览器访问模式。

### 2.2 关键决策问题

| 问题 | 结论 | 证据/原因 |
|------|------|-----------|
| 是否继续使用前端存储 | 否 | 当前所有会话被序列化到一个 `localStorage` key，容量、同步写入和内存复制均不适合大历史。 |
| 是否使用单个完整 JSON 文件 | 不推荐 | 每新增一条消息都需要读取/序列化/覆盖整个文件，1M Token 会产生明显写放大和崩溃恢复风险。 |
| 是否一个会话一个物理对象 | 是 | 每个会话使用独立目录，目录内按大小分段；删除、压缩、恢复和并发隔离更简单。 |
| 是否压缩正在写入的数据 | 否 | 活跃分段直接追加，避免每次写入重压缩；分段封存后在后台压缩。 |
| 是否一次加载完整会话 | 否 | 历史列表只取元数据，消息按游标倒序分页，前端仅保留当前可见窗口。 |
| 1M Token 是否等于每次都发送给模型 | 否 | 持久化容量与模型上下文窗口是两个概念；实际发送量必须受当前模型上下文上限约束。 |

### 2.3 范围与非范围

- 范围：前端历史仓储接口、后端会话文件存储、REST API、分页、压缩、配额、迁移、异常恢复、上下文组装。
- 非范围：云同步、多用户账号同步、全文语义检索、跨设备同步、数据库引入。

## 3. 当前实现与差距

### 3.1 当前技术栈

| 层次 | 技术 | 证据 |
|------|------|------|
| 前端 | React + TypeScript + Vite | `DevBridge-Front/package.json` |
| 后端 | Java 17 + Spring Boot 3.3.7 + Jackson | `DevBridge-Server/pom.xml` |
| 桌面端 | Electron，后端由客户端本地启动 | `DevBridge-Electron/src/main.js` |
| AI 数据目录 | Electron 已使用 `app.getPath('userData')/runtime/ai` | `DevBridge-Electron/src/main.js` |

### 3.2 当前历史存储限制

当前 `AiChatPanel.tsx` 采用以下策略：

- 所有会话存入 `devbridge.ai.conversations.v1` 一个 `localStorage` key。
- 最多 30 个会话。
- 每个会话最多持久化 50 条消息。
- 单条持久化正文最多 8000 字符。
- 单个工具输出最多 2500 字符。
- 全部历史序列化结果最多约 100 万字符。
- 下一轮模型上下文只发送最近 12 条普通文本，每条最多 1200 字符。

这意味着当前实现既不能满足 1000 会话，也不能满足 1M Token 的持久化和上下文连续性。同步 `localStorage.setItem(JSON.stringify(...))` 还会在渲染进程中制造大对象复制、长任务和内存峰值。

## 4. 推荐架构

### 4.1 模块边界

新增独立 `ai.history` 后端模块，和 Provider、ADB 工具、Local Shell 工具解耦：

```text
AiChatPanel
  -> ConversationRepository（前端接口）
      -> HttpConversationRepository
          -> /api/ai/conversations
              -> ConversationApplicationService
                  -> ConversationStore（接口）
                      -> SegmentedFileConversationStore
```

约束：

- UI 不直接访问文件系统，普通浏览器与 Electron 统一通过后端 API。
- AI Provider 不负责历史存储。
- MCP 工具只产出工具结果，历史模块只保存标准化消息快照。
- `AiConversationService` 通过 `conversationId` 获取上下文，不依赖前端回传完整历史。
- 保留现有请求中的 `history` 一段兼容期，完成迁移后再废弃。

### 4.2 文件布局

推荐目录：

```text
<ai-history-root>/
  manifest.json
  conversations.index.json
  sessions/
    <conversation-id>/
      meta.json
      000001.jsonl.gz
      000002.jsonl.gz
      000003.jsonl
```

- `conversations.index.json`：只保存历史列表需要的轻量字段，不保存消息正文。
- `meta.json`：标题、创建时间、更新时间、消息数、字符数、估算 Token 数、总字节数、最后序号和格式版本。
- `*.jsonl`：当前活跃追加写分段。
- `*.jsonl.gz`：达到阈值或会话进入冷状态后封存压缩的只读分段。
- 每个分段建议上限 4 MiB 或 200 条完整消息，任一达到即轮转。阈值应可配置。

使用会话目录而不是强制单文件，是为了让追加写、分页、压缩、恢复和大会话读取都保持有界成本。业务上仍然是“一次对话一个独立存储单元”。

### 4.3 JSONL 记录模型

每行是一条完整、可独立解析的记录：

```json
{"schemaVersion":1,"sequence":42,"messageId":"...","role":"assistant","kind":"text","content":"...","createdAt":"...","process":null,"checksum":"..."}
```

设计原则：

- 用户消息发送成功前先落盘。
- AI 流式增量不逐 Token 写盘，避免几千次小 IO。
- AI 完成回复后一次追加完整消息；长回复可每 1 秒或 64 KiB 生成恢复检查点，但检查点不进入正常历史列表。
- 工具执行过程在步骤完成后写入，不持久化每个 UI 动画状态。
- `sequence` 在单会话内单调递增，用于稳定排序和游标分页。
- `messageId` 作为幂等键，防止网络重试导致重复消息。
- 校验和用于启动扫描时识别尾部半条记录；损坏尾部可截断到最后一个有效换行，不影响前面记录。

### 4.4 写入策略

1. 单会话使用轻量锁，不同会话可以并行写入。
2. 使用 `FileChannel` 或带缓冲的输出流追加，禁止覆盖整个会话文件。
3. 用户消息、最终 AI 回复、确认结果等业务边界完成后 `flush`。
4. 不对每个 SSE chunk 调用 `fsync`；建议在最终消息、应用退出和定时检查点调用 `force(false)`。
5. `meta.json` 和全局索引通过“临时文件写入 + 原子 rename”更新，避免断电留下半文件。
6. 索引更新去抖，流式回复期间不反复重写索引。

### 4.5 压缩策略

- 首选 JDK 内置 GZIP，避免新增依赖；使用偏速度的压缩级别。
- 活跃分段不压缩，封存分段异步压缩，压缩完成后原子替换。
- 小于 64 KiB 的分段不压缩，避免文件头和 CPU 成本得不偿失。
- 打开历史会话时只解压所需分段，不把整个会话解压进内存。
- 对文本、JSON、日志类内容，预计压缩率通常为 2:1 到 6:1，实际值必须用真实工具输出验证。

### 4.6 查询与分页

- 历史列表：基于 `updatedAt + id` 游标分页，每页建议 50 条。
- 消息列表：基于 `sequence` 倒序分页，首次只加载最后 50 条。
- 用户上滚时加载更早消息；切换会话时释放上一会话的大消息对象。
- 前端不再持有 1000 个会话的完整 `messages`，只持有列表摘要和当前会话的有限消息窗口。
- 超长工具输出默认在卡片中展示摘要，需要时按独立接口懒加载完整内容。

### 4.7 建议 API

| 方法 | 路径 | 用途 |
|------|------|------|
| `GET` | `/api/ai/conversations?cursor=&limit=` | 分页查询会话摘要 |
| `POST` | `/api/ai/conversations` | 新建会话 |
| `PATCH` | `/api/ai/conversations/{id}` | 重命名或更新状态 |
| `DELETE` | `/api/ai/conversations/{id}` | 删除会话，先移动到回收目录再异步清理 |
| `GET` | `/api/ai/conversations/{id}/messages?before=&limit=` | 倒序分页读取消息 |
| `POST` | `/api/ai/conversations/{id}/messages` | 幂等追加完整消息 |
| `GET` | `/api/ai/conversations/storage` | 查询磁盘占用、配额和压缩状态 |
| `POST` | `/api/ai/conversations/import-local` | 一次性导入旧 `localStorage` 历史 |

## 5. 1M Token 支持定义

### 5.1 持久化容量

建议将验收口径定义为：

- 单个会话可保存至少 1,000,000 个估算 Token，不截断用户和 AI 正文。
- 会话记录按流读取和分页返回，任何一次 API 调用不要求把 1M Token 全部加载到前端。
- 存储元数据同时记录精确字节数、字符数和估算 Token 数。
- 不用固定字符数代替 Token 上限；不同语言和模型分词差异较大。

粗略容量估算：1M Token 的纯文本一般约为 3 至 8 MiB，工具日志、JSON 转义和元数据可能显著放大。1000 个会话如果都达到 1M Token，原始数据可能达到数 GiB，压缩后仍需预留约 1 至 4 GiB 或更多空间。因此“支持 1000 会话”不能等同于“默认永久保存任意体积而不设磁盘策略”。

### 5.2 模型运行时上下文

存得下 1M Token，不代表任意模型一次请求都能接收 1M Token。建议上下文组装器按以下顺序构建请求：

1. 系统提示词和工具定义。
2. 最近若干轮完整消息。
3. 当前任务相关的历史消息或工具结果。
4. 较早历史的分层摘要。
5. 为模型输出预留 Token 空间。

后端根据当前模型配置的上下文窗口设置预算。如果模型本身支持 1M Token，可以提高读取预算；如果模型只支持 32K/128K，则必须摘要或检索，不能强行发送 1M Token。当前前端“最近 12 条、每条 1200 字符”的固定截断应由后端预算式上下文组装替代。

## 6. 磁盘与性能策略

### 6.1 配额

- 默认建议全局历史配额 5 GiB，可在 AI 设置中调整。
- 80% 时提示用户并触发更积极的冷数据压缩。
- 95% 时停止创建恢复检查点和非必要过程明细，但不得截断最终业务回复。
- 100% 时明确返回“历史存储空间不足”，不能静默删除历史或让 AI 对话假装保存成功。
- 自动清理必须由用户显式开启，可按最旧会话、最大保留天数或回收站大小执行。

### 6.2 内存

- 历史列表只加载摘要。
- 消息按页加载，后端使用流式 JSON 解析和有界缓冲。
- 压缩/解压以分段为单位，单次工作集受 4 MiB 分段阈值约束。
- 前端虚拟化长消息列表；不可把 1M Token 会话全部放进 React state 或一次 Markdown 渲染。

### 6.3 IO

- 正常消息保存是顺序追加，单条写入复杂度接近 O(消息大小)。
- 更新索引不跟随 SSE chunk，只在会话元数据真正变化后去抖更新。
- 压缩任务单线程、低优先级执行，避免与 ADB 日志采集和 AI SSE 争抢磁盘与 CPU。
- 应用退出前等待当前消息提交和索引原子替换完成，但设置合理超时。

## 7. 存储路径与运行模式

- Electron：建议 `<app.getPath('userData')>/runtime/ai/history`。
- 独立后端：新增 `devbridge.ai-history-root`，默认 `target/devbridge-ai/history`，生产启动时可显式指向用户数据目录。
- Electron 启动后端时显式传入 `--devbridge.ai-history-root=...`。
- 不建议让前端直接通过 Electron IPC 写文件，否则浏览器模式和 Electron 模式会产生两套实现。

## 8. 安全与隐私

- 历史可能包含设备标识、日志、文件路径、命令输出和用户隐私，目录权限应限制为当前操作系统用户可读写。
- API 仅绑定 `127.0.0.1`，并校验会话 ID，禁止 `../` 等路径穿越。
- 文件名只能使用服务端生成的 UUID，不使用用户标题。
- 日志中不得输出完整对话、工具 stdout、API Key 或文件原文。
- 如果需要静态加密，采用“先压缩、后 AES-GCM 加密”的封存分段；密钥沿用现有本机密钥管理思路。加密会增加恢复和排障成本，建议作为可配置能力并单独验收。
- 删除操作先移动到同文件系统回收目录，短暂保留后再物理删除，避免误删和半删除状态。

## 9. 旧数据迁移

1. 新版本首次打开 AI 面板时检测旧 key `devbridge.ai.conversations.v1`。
2. 前端将旧数据分批提交给导入接口，服务端按会话和消息幂等写入。
3. 服务端返回导入会话数、消息数和失败项。
4. 前端重新读取后端列表并核对成功后，才删除旧 `localStorage` key。
5. 迁移失败保留旧数据，允许下次重试，不阻断新对话。

由于旧数据已经被截断，迁移只能保留现有内容，无法恢复此前被 8000/2500 字符限制丢弃的正文或工具输出。

## 10. 候选方案对比

| 方案 | 写入性能 | 大会话读取 | 压缩 | 崩溃恢复 | 复杂度 | 结论 |
|------|----------|------------|------|----------|--------|------|
| 单会话单个完整 JSON 覆盖 | 差，写放大 O(会话总量) | 必须整文件解析 | 可压缩但每次需重压 | 较差 | 低 | 不推荐 |
| 单会话单个无限 JSONL | 好 | 尾部分页和压缩管理较难 | 活跃文件难压缩 | 好 | 低 | 小规模可用，不满足长期目标 |
| 会话目录 + 分段 JSONL + 冷压缩 | 好，顺序追加 | 有界分页 | 好 | 好 | 中 | 推荐 |
| SQLite/其他嵌入式数据库 | 好 | 好 | 依赖数据库机制 | 好 | 中 | 用户已明确排除 |

## 11. 风险矩阵

| 风险 | 等级 | 触发条件 | 影响 | 缓解措施 | 回退方案 | PoC |
|------|------|----------|------|----------|----------|-----|
| 1M Token 被误解为每次全量发给模型 | 高 | 模型窗口小于历史量 | Provider 拒绝、成本和延迟失控 | 区分存储容量与请求预算，按模型窗口组装上下文 | 回退到最近消息 + 摘要 | 是 |
| 长会话重新引发 Electron OOM | 高 | 前端一次加载或渲染全部历史 | 页面崩溃 | 分页、虚拟化、懒加载工具输出 | 降低单页大小 | 是 |
| 流式期间高频落盘 | 高 | 每个 SSE chunk 都保存 | IO 放大、闪烁、卡顿 | 完整消息提交，低频恢复检查点 | 关闭检查点 | 是 |
| 尾部记录因异常退出损坏 | 中 | 写入中进程退出 | 最后一条消息不完整 | 换行记录、校验和、启动修复 | 截断到最后有效记录 | 是 |
| 压缩任务影响实时日志和 AI | 中 | 大量冷数据同时压缩 | CPU/磁盘争用 | 单线程低优先级、限速、空闲执行 | 暂停压缩 | 是 |
| 索引与会话文件不一致 | 中 | 原子替换前异常退出 | 列表缺失或计数错误 | 索引可由 `meta.json` 重建 | 启动后台重建索引 | 是 |
| 磁盘配额耗尽 | 中 | 工具输出或日志长期增长 | 保存失败 | 配额、预警、用户可控清理 | 禁止新增并明确报错 | 是 |
| 旧历史迁移重复 | 低 | 导入重试 | 重复会话/消息 | UUID 和 messageId 幂等 | 清理单次迁移批次 | 否 |

## 12. PoC 验证建议

建议用隔离测试目录生成数据，不接入生产 UI，验证：

1. 创建 1200 个会话，列表首屏读取 P95 小于 100 ms。
2. 单会话写入至少 1M Token，进程内存峰值不随总会话体积线性增长。
3. 追加一条 10 KiB 消息 P95 小于 20 ms，不包含强制磁盘同步的极端硬件耗时。
4. 首次加载最后 50 条消息 P95 小于 100 ms。
5. GZIP 压缩后体积、CPU 时间和前台功能延迟达到可接受范围。
6. 在记录写入、索引替换和压缩替换三个阶段强制终止进程，重启后可以恢复到最后一条完整记录。
7. 前端连续上滚加载长会话，Electron 渲染进程内存保持有界且 Markdown 不重复解析全部历史。

指标必须在 macOS Electron 和普通浏览器访问两种模式分别测试；两者共享后端实现，但渲染进程内存行为不同。

## 13. 待确认项

需要确认“至少支持 1M Token”的口径：

- 推荐口径：单个会话可持久化并检索 1M Token；每次实际发送给模型的上下文量由模型窗口决定。
- 如果要求任意模型每次请求都携带完整 1M Token，则技术上无法由本地存储单方面保证，必须限定支持 1M 上下文的模型，并接受显著的请求成本和延迟。
