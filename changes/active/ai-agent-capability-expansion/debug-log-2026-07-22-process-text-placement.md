# 思考过程文本位置错误排查记录

## 排查步骤

- [x] 根据截图确认异常文字位于过程卡片之后、最终回复之前。
- [x] 检查前端 SSE 工具事件消费顺序。
- [x] 验证第一次工具调用前文本会转换为过程消息，后续工具调用前文本不会再次归并。
- [x] 修正后续过程文本归并逻辑，并保持最后一次工具调用后的正式总结不变。
- [x] 根据再次出现的闪现截图检查归并发生前的页面渲染时序。
- [x] 验证工具事件到达前，临时 Assistant 消息已经作为普通回复参与页面绘制。
- [x] 增加工具流程后文本的待判定状态，并补充模型工具调用输出约束。
- [x] 复测发现待判定直到流结束会让最终回复退化为一次性展示。
- [x] 引入明确的最终回复内部标记，恢复标记后正文的实时流式渲染。

## 根因

`streamAssistantTurn` 只在首次创建过程卡片时调用 `convertAssistantToProcess`。过程卡片创建后，模型在两个工具调用之间输出的新说明会创建普通 Assistant 消息；下一次工具事件仅追加工具步骤并清空消息标识，没有把该消息转入既有过程，因此这些说明被遗留在最终回复前面。

后续复测发现，即使下一次工具事件能够正确删除并归并这条消息，事件到达前该消息仍会作为普通 Assistant 回复渲染一帧或多帧，形成截图中的“一闪而过”。根因是工具事件边界确认晚于文本 chunk，前端此前没有区分“工具流程中的待判定文本”和“已确认最终回复”。

第一次修复把工具后的文本隐藏到流结束，虽然消除了闪现，却把最终回复也延迟到完成事件后一次性展示。根因是“是否还会调用工具”不能通过等待时间或连接完成状态同时满足过程隔离与实时输出，必须由模型输出协议提供明确边界。

## 修复

- 工具事件到达时，将最终标记前积累的模型说明追加到现有过程卡片。
- 模型最终回复使用内部 `<DEVBRIDGE_FINAL>` 标记划定边界，标记前文本进入过程，标记后正文立即进入原有平滑流式渲染管线。
- 后端持久化前移除内部标记及其前置过程文本，历史记录和断流恢复只保留最终业务正文。
- 未支持标记的 Provider 在流结束时兼容回退为最终正文，保证业务结果不丢失。
