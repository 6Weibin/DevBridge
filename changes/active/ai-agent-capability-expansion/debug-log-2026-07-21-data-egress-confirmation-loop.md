## 根因定位链

1. 用户批准的数据外发确认严格绑定上一次工具结果摘要，该安全约束正确。
2. 旧续跑逻辑只恢复 `stepId` 和 `modelCallId`，没有恢复已完成工具的 `toolCallId`。
3. 模型续跑后再次请求 `device.list`，后端为其生成新 `toolCallId` 并重新执行。
4. 工具结果包含执行耗时和调用来源标识，每次结果字节不同，`dataDigest` 因此每轮改变。
5. 原确认无法匹配新摘要，后端不断创建新确认，形成无限循环。

第二次现场验证还发现了独立的确认状态错误：工具调用标识和摘要已经稳定，但 Runtime 在续跑前将确认从 `ACCEPTED` 更新为 `CONSUMED`，数据外发守卫却只认可 `ACCEPTED`，因此仍然创建新确认。

## 修复方案

- 数据外发确认续跑时，从 Checkpoint 恢复同一步骤中已成功工具的工具标识、参数摘要和原 `toolCallId`。
- 对已经循环过的旧任务，根据有序 Agent Event 选择当前确认前最近完成的工具调用，不会误重放更早结果。
- 模型再次请求相同工具和参数时，Tool Gateway 通过原幂等键重放持久化结果，不再真实执行。
- 重放结果与用户批准时完全一致，原数据摘要可精确命中，不放宽确认的安全边界。
- 外发守卫允许 `CONSUMED` 确认完成其正在进行的精确绑定续跑；任务、步骤、模型调用、Provider、模型、数据摘要和过期时间仍必须全部一致。
