# 流式回复渲染卡顿排查记录

## 排查步骤

- [x] 检查 SSE 消费、前端 chunk 缓冲、Markdown 分段和自动滚动链路。
- [x] 验证固定 `240ms` 刷新会把连续 chunk 合并为每秒约 4 次的跳变。
- [x] 验证 `useDeferredValue` 对流式尾段形成第二层延迟。
- [x] 检查流式 Markdown 的同步布局测量，确认每次内容更新都会触发 `getBoundingClientRect()`。
- [x] 完成平滑刷新、即时尾段解析和流式布局测量收敛。
- [x] 完成前端构建、1M Token 分段测试和 Electron Renderer 内存测试。

## 根因

SSE 本身持续返回数据，但 `useAiChatStore` 固定等待 `240ms` 才把累计文本写入 React 状态，视觉刷新率只有约 4 FPS。随后 `MarkdownSegment` 又通过 `useDeferredValue` 延迟解析，`LazyMarkdownSegment` 还在每次尾段变化后同步测量 DOM 高度，三者叠加后表现为正文成段跳出和页面短暂停顿。

## 修复方案

1. 将流式刷新调整为约 30 FPS，并根据待显示字符量自适应释放 32 至 2048 个字符。
2. 网络大 chunk 分多个刷新周期显示，普通小 chunk 及时进入页面；完成和异常边界仍立即写完剩余数据，保证回复完整。
3. 流式尾段直接解析 Markdown，不再叠加延迟更新。
4. 流式期间跳过无业务价值的同步高度测量，回复完成后再记录最终高度。
5. 保留 6000 字符稳定分段、12000 字符可变尾段和屏外懒挂载，不恢复整篇 Markdown 重解析。

## 验证结果

- Vite 生产构建通过。
- 4,000,000 字符约 1M Token 内容完整，形成 668 个分段，最大分段 5997 字符。
- Electron Renderer 仅挂载 16 个分段，私有内存约 293MB，未发生崩溃。
- Electron 使用最新资源启动，后端首页返回 HTTP 200。

## 长任务回归修正

首次方案采用约 30 FPS，并把小 chunk 继续拆成多个刷新周期。实际长任务中，这会产生多于 SSE 事件数量的 React 提交，Markdown 解析和自动滚动布局读取持续占用 Renderer 主线程，最终表现为全屏白屏和无响应。

修正后采用约 12 FPS 的有界刷新：192 字符以内一次写入，只有大块积压才按比例拆分；超过 2000 字符的流式 Markdown 尾段使用低优先级解析。该策略不减少回复内容，只限制单位时间内的渲染工作量。

## Electron OOM 根因修正

实际复现日志确认 Renderer 在约 7 分钟后达到 4GB V8 Heap 并 OOM。进一步定位发现首次平滑输出实现使用单个字符串执行 `buffer += chunk` 和重复 `slice()`，V8 会形成并保留 Rope/Sliced String 关联，已消费正文的底层存储不能及时释放。

最终实现改为 chunk 引用队列和消费游标：接收时只保存原始 chunk 引用，刷新时按完整 chunk 移动游标，定期仅压缩数组引用，不再对持续增长的正文字符串做头部切片。该调整消除了逻辑缓冲很小但 V8 Heap 持续增长的问题。
