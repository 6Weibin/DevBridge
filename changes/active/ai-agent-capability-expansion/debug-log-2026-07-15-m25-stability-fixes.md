- [x] 跟踪确认状态从 `PENDING -> ACCEPTED -> CONSUMED` 到模型续跑的完整调用链。
- [x] 检查应用启动生命周期，确认原实现没有非终态任务扫描入口。
- [x] 检查前端取消异常分支，确认接口失败后仍会本地 abort 并显示取消成功。
- [x] 检查流式正文和工具事件顺序，确认 `finalAnswer` 在工具事件后没有重新划定边界。
- [x] 检查重复批准终态路径，确认任务版本高于旧 Checkpoint 时会恢复失败。

## 假设与验证

| 假设 | 验证结果 |
|---|---|
| Checkpoint 可读取就等于任务可自动恢复 | 不成立；原实现只有批准 HTTP 请求会调用续跑服务 |
| 前端 abort 可以代表后端取消成功 | 不成立；取消接口异常被吞掉，后台任务可能继续运行 |
| `finalAnswer` 只包含工具后的最终回答 | 不成立；所有模型 chunk 都持续追加，工具事件没有重置 |
| `CONSUMED` 确认可以无限次幂等批准 | 不成立；任务完成后 Checkpoint 版本落后，会被一致性校验拒绝 |

## 根因链

1. 恢复数据已经持久化，但缺少启动调度入口，导致控制面能力没有形成运行闭环。
2. 前端把“停止本地订阅”和“后端任务取消成功”当成同一结果。
3. 前端把工具前文本转为过程卡片，但后端没有同步采用相同的最终回复边界。
4. 重复确认仍按运行中恢复处理，没有先判断任务是否已经进入终态。

## 修复方案

- 复用 `AgentRuntimeContinuationService` 增加启动分页扫描，只恢复带已批准确认关联的确定性续跑任务。
- 后台恢复丢弃兼容 SSE 展示正文，结果继续由 Agent Task 加密持久化，避免无人订阅时积累内存。
- 终态重复确认直接重放任务结果，不重新执行工具。
- 工具事件到达时清空工具前回复缓存，只持久化工具后的最终内容。
- 前端校验取消接口返回状态；失败时保留当前流并展示详细错误。
