## 2. 背景

Ai DevBridge 当前已经具备以下核心功能：

- Android、iOS 等设备发现和设备信息展示。
- 文件浏览、详情查看、预览、复制、重命名、下载和删除。
- Android、iOS 实时日志采集、过滤、停止和导出。
- Android 应用列表、应用详情和应用卸载。
- Bridge Copilot 普通对话、流式回复、Markdown 渲染和历史聊天界面。
- OpenAI、DeepSeek、Qwen、GLM、ERNIE 和 OpenAI-compatible Provider 配置。
- ADB 工具调用能力。
- 本机终端工具调用能力。
- 敏感操作确认和命令安全等级控制。
- 基础 AI Provider 调用观测。

当前 AI 助手已经能够根据用户问题调用 ADB 或本机终端工具，但整体仍属于支持工具调用的单 Agent 助手。后续需要围绕设备管理、日志分析、应用诊断、文件处理和本地开发环境，逐步建设可持久化、可编排、可恢复、可观测的 Agent 系统。

## 3. 建设目标

1. 让 AI 从“回答问题”升级为“完成设备诊断和操作任务”。
2. 让复杂任务可以拆分为多个可观察、可恢复的执行步骤。
3. 让设备、日志、应用、文件和本机操作使用独立、结构化的工具边界。
4. 让历史聊天、任务状态、设备状态和故障经验形成可持续使用的 Memory。
5. 让 AI 可以检索本地文档、故障案例、操作规范和设备知识。
6. 让复杂问题由多个专业 Agent 协同分析，并由验证 Agent 检查结果。
7. 保持 AI 模块与现有设备管理功能解耦，避免对主业务页面进行侵入式修改。

## 4. 当前能力基线

| 能力 | 当前状态 | 已有实现 | 主要问题 | 后续定位 |
|------|----------|----------|----------|----------|
| 普通 AI 对话 | 已实现 | 支持流式输出、Markdown、取消和错误展示 | 长期上下文和任务状态不足 | 作为 Agent 交互入口 |
| Tool Calling Agent | 已实现 | 模型可调用 ADB 和本机终端工具 | 主要依赖模型隐式循环，缺少任务状态和循环治理 | 升级为统一工具执行层 |
| ADB 工具 | 已实现 | 覆盖设备、网络、端口、文件、Shell、应用、日志和设备控制 | 工具粒度偏底层，部分业务仍依赖原始命令 | 保留底层工具并增加领域工具 |
| 本机终端工具 | 已实现 | 支持执行命令、查看目录、读取文本、查询和取消进程 | 本机开发任务缺少工作流和上下文隔离 | 作为 Local Agent 执行层 |
| 敏感操作确认 | 已实现 | 支持低、中、高安全等级和用户确认 | 确认前后任务恢复仍依赖提示词续跑 | 纳入 Agent Checkpoint 状态机 |
| 历史聊天 | 部分实现 | 支持新建、选择、重命名和删除历史聊天 | 数据在 `localStorage`，容量和上下文有限 | 改为本地文件持久化 Memory |
| 对话上下文 | 部分实现 | 最近 12 条普通文本进入下一轮请求 | 单条和总长度被截断，工具过程没有形成可靠记忆 | 使用 Token 预算和分层摘要 |
| 固定工作流 | 初步实现 | 日志分析和敏感操作确认存在少量固定逻辑 | 流程散落，无法统一暂停、恢复和重试 | 建设独立 Workflow Runtime |
| 动态任务路由 | 初步实现 | 根据关键词选择 ADB 或本机终端范围 | 容易误判，无法路由到专业 Agent | 使用结构化 Router Agent |
| RAG | 未实现 | 后端只有能力边界占位 | 无文档索引、故障库和检索链路 | 建设本地知识库 |
| 多 Agent | 未实现 | 当前由一个模型承担理解、执行和回答 | 复杂问题缺少分工、验证和汇总 | 引入 Supervisor 和专业 Agent |
| Agent 可观测性 | 部分实现 | 记录 Provider、模型、耗时和成功状态 | 缺少步骤、工具、Token、成本和任务轨迹 | 建设 Agent Trace 和指标体系 |

## 5. Agent 能力说明

| 能力 | 定义 | 在 Ai DevBridge 中的作用 |
|------|------|---------------------------|
| Tool Calling Agent | 模型根据任务选择并调用工具，将工具结果继续交给模型分析 | 执行设备查询、ADB 命令、本机命令、日志采集和文件操作 |
| 固定工作流编排 | 按预定义步骤、条件和异常分支执行任务 | 设备健康检查、连接诊断、日志分析、安装卸载和文件备份 |
| 动态任务路由 | 根据问题、设备平台、风险和上下文选择 Agent、模型和工具 | 将任务路由到设备、日志、应用、文件或本机开发 Agent |
| Memory | 保存聊天、任务、设备、故障和用户偏好信息 | 支持连续对话、任务恢复、设备趋势和历史故障对比 |
| RAG | 从知识库检索相关文档和历史案例后辅助模型回答 | 查询错误码、ADB 文档、已知问题、操作手册和历史解决方案 |
| MCP/领域工具 | 用统一契约暴露设备、日志、文件、应用和本机能力 | 解耦 Agent 与现有业务实现，并控制权限和风险 |
| 多 Agent | 多个具备不同职责的 Agent 协同完成任务 | 分别负责采集、诊断、验证、知识检索和报告生成 |

| 移动端测试 Agent | 多 Agent、Workflow | 自动完成安装、启动、基础操作、日志检查、截图和结果报告 | 测试规划 → 安装 → 执行 → 采集 → 验证 → 报告 | 应用、ADB、截图和日志工具 | 测试设备隔离；禁止默认执行破坏性操作 | P1 |

| 自动故障报告 | Workflow、Memory | 将设备信息、日志证据、工具命令和分析结果整理成标准报告 | 收集 → 脱敏 → 生成 → 用户预览 → 导出 | 会话持久化和报告模板 | 导出前执行敏感信息检查 | P1 |
| 设备诊断基线库 | Memory、RAG | 保存不同型号、系统和应用版本的正常指标及典型故障 | 正常数据采集 → 建立基线 → 异常对比 → 趋势分析 | 设备快照和本地索引 | 数据容量、设备标识脱敏和基线版本控制 | P2 |
| 跨平台设备 Agent | 动态路由、MCP | 根据 Android、iOS 和 HarmonyOS 平台选择对应工具集 | 平台识别 → 能力发现 → Agent 路由 → 平台工具执行 | ADB、iOS 和 HDC 领域工具 | 工具必须绑定平台，禁止跨平台误调用 | P1 |

## 8. 领域 MCP 规划

现有 ADB 和本机终端工具应继续保留，但上层 Agent 应优先调用结构化领域工具，只有领域工具无法覆盖时才回退到原始命令。

| MCP/工具域 | 主要能力 | 复用现有功能 | 新增需求 | 风险等级 |
|------------|----------|--------------|----------|----------|
| Device MCP | 设备列表、详情、健康指标、截图、连接诊断 | 设备发现、详情、截图、ADB诊断 | 健康快照、平台能力发现 | 低至中 |
| Log MCP | 启动采集、停止采集、读取窗口、过滤、导出、异常摘要 | 实时日志和日志导出 | AI 独立采集会话、日志窗口和事件模型 | 低至中 |
| App MCP | 应用列表、详情、安装、卸载、启动、停止、权限查询 | 应用列表、详情和卸载 | 安装、启动停止、权限和签名检查 | 中至高 |
| File MCP | 目录、详情、搜索、预览、复制、下载、删除、重命名 | 文件管理接口 | 条件搜索、批量计划和校验 | 低至高 |
| ADB MCP | ADB 全部底层能力和任意受控 Shell | 已实现 | 动态工具发现和更细粒度观测 | 低至高 |
| Local Shell MCP | 本机命令、目录、文件、进程 | 已实现 | 构建任务、开发环境诊断和工作区隔离 | 低至高 |
| Knowledge MCP | 文档搜索、故障案例、错误码和操作规范查询 | 无 | 本地知识索引和引用返回 | 低 |
| Report MCP | 创建和导出 Markdown、JSON、ZIP 等报告 | 日志 ZIP 导出 | 故障报告、诊断包和脱敏检查 | 低至中 |

## 10. 固定工作流规划

| 工作流 | 触发方式 | 主要步骤 | 可暂停点 | 结束条件 | 优先级 |
|--------|----------|----------|----------|----------|--------|
| 设备健康检查 | 用户请求或快捷入口 | 获取设备 → 采集指标 → 判断异常 → 补充检查 → 生成报告 | 无设备、需要敏感命令 | 健康报告生成或设备断开 | P0 |
| 实时日志分析 | 用户请求 | 启动采集 → 收集窗口 → 检测异常 → 补充诊断 → 停止采集 → 报告 | 用户停止、敏感操作 | 达到采集时间、证据充分或用户取消 | P0 |
| ADB 连接诊断 | 设备无法识别时 | 检查工具 → 检查服务 → 枚举设备 → 检查授权 → 修复建议 | 重启 ADB 服务 | 设备可用或输出明确原因 | P0 |
| 应用故障诊断 | 用户指定包名 | 查询应用 → 获取日志 → 检查权限 → 检查资源 → 汇总结论 | 需要启动/停止应用 | 结论通过验证或无法继续采集 | P0 |
| 文件备份 | 用户选择范围 | 检查空间 → 计算大小 → 拉取 → 校验 → 压缩 → 报告 | 大文件、目标覆盖 | 文件校验通过或任务取消 | P1 |
| 应用安装卸载 | 用户发起 | 前置检查 → 风险确认 → 执行 → 验证 → 失败补偿 | 安装、卸载确认 | 操作验证成功或输出失败原因 | P1 |
| 本机构建验证 | 用户请求 | 检查环境 → 执行构建 → 执行测试 → 分析错误 → 报告 | 依赖下载、敏感脚本 | 构建测试完成或用户取消 | P1 |

| Memory 类型 | 保存内容 | 作用范围 | 建议存储方式 | 生命周期 |

| Task Memory | 任务目标、计划、步骤、工具结果、确认状态、Checkpoint | 单次 Agent 任务 | 独立任务状态文件 | 任务完成后归档 |

| Working Memory | 当前模型调用所需最近消息、摘要和检索结果 | 单次模型请求 | 内存中按 Token 预算组装 | 请求结束后释放 |

## 13. 动态任务路由规则

路由结果应为结构化数据，不应只依赖关键词匹配。

| 路由维度 | 示例 | 路由结果 |
|----------|------|----------|
| 业务领域 | 日志、应用、文件、设备、本机开发 | 选择对应专业 Agent |
| 设备平台 | Android、iOS、HarmonyOS | 选择对应平台工具集 |
| 操作类型 | 查询、分析、修改、删除、安装、卸载 | 决定工具权限和确认策略 |
| 任务复杂度 | 单工具查询、多步骤诊断、多设备并行 | 选择 Tool Agent、Workflow 或 Multi-Agent |
| 模型能力 | 普通文本、工具调用、多模态、长上下文 | 选择合适 Provider 和模型 |
| 风险等级 | 低、中、高 | 直接执行、用户确认或阻断 |
| 上下文完整性 | 包名缺失、设备未选择、目标不明确 | 先向用户澄清，不执行工具 |

## 14. Agent 任务状态

| 状态 | 说明 | 可执行动作 |
|------|------|------------|
| CREATED | 任务已创建，尚未规划 | 开始规划、取消 |
| PLANNING | 正在分析目标并生成计划 | 更新计划、失败、进入执行 |
| RUNNING | 正在执行步骤或工具 | 继续、暂停、失败、完成 |
| WAITING_CONFIRMATION | 等待用户确认敏感操作 | 确认、拒绝、超时取消 |
| WAITING_INPUT | 缺少设备、包名、路径等必要输入 | 用户补充、取消 |
| PAUSED | 用户主动暂停或应用准备退出 | 恢复、取消 |
| RETRYING | 可重试步骤正在重试 | 返回执行、失败 |
| COMPLETED | 任务已完成并生成结果 | 查看报告、归档 |
| FAILED | 任务失败且包含明确原因 | 重试、重新规划、结束 |
| CANCELED | 用户取消任务 | 查看已完成步骤 |

## 15. 可观测性需求

| 观测对象 | 需要记录 | 不允许记录 |
|----------|----------|------------|
| Agent 任务 | 任务 ID、Agent、状态、步骤、开始结束时间 | 完整敏感对话正文 |
| 模型调用 | Provider、模型、输入输出 Token、耗时、完成原因、成功状态 | API Key、Authorization Header |
| 工具调用 | 工具名、风险等级、命令摘要、状态、耗时、退出码 | 未脱敏的敏感输出 |
| 工作流 | 当前节点、重试次数、暂停原因、恢复次数 | 用户隐私数据原文 |
| RAG | 检索耗时、命中文档、分数、文档版本 | 未授权文档全文日志 |
| 多 Agent | 路由结果、Agent 分工、汇总结果、验证状态 | 模型私有推理过程 |

## 16. 安全要求

1. 所有工具调用必须经过工具白名单、参数校验和风险分级。
2. 删除、卸载、覆盖、系统修改和高风险本机命令必须遵循用户授权配置。
3. Agent 不得通过重新描述、拆分命令或切换工具绕过确认策略。
4. 每次用户确认必须绑定任务、步骤、工具、参数摘要和有效期。
5. 已确认并完成的工具必须记录幂等键，恢复任务后不得重复执行。
6. 文件路径必须规范化，禁止路径穿越和访问未授权目录。
7. 日志、文件、截图、位置和设备标识进入模型前必须经过脱敏策略。
8. API Key 不进入前端持久化、任务历史、Agent Trace 或错误详情。
9. Agent 循环必须配置最大步骤数、最大工具调用次数、Token 预算和总超时。
10. 高风险任务失败时默认停止，不允许模型自行降低安全等级继续执行。

## 17. 非功能需求

| 类型 | 要求 |
|------|------|
| 解耦 | Agent、Workflow、Memory、RAG 和领域工具必须位于独立模块，不把业务逻辑写入现有页面主体 |
| 性能 | 流式回复、日志采集和 Agent 工具事件不能阻塞 UI；历史消息必须分页加载 |
| 内存 | 不允许把完整 1M Token 会话加载进 React 状态或一次性 Markdown 渲染 |
| IO | 会话和任务状态使用追加写、分段、压缩和原子替换，避免整文件反复覆盖 |
| 可恢复 | 敏感确认、长任务和应用退出场景需要有 Checkpoint 和幂等恢复 |
| 可取消 | 模型调用、日志采集、外部命令和工作流必须支持用户取消 |
| 容错 | 单个工具失败不能导致整个页面崩溃；工作流根据策略重试、降级或停止 |
| 可观测 | 每个任务和步骤具有统一 ID，能够关联模型调用、工具调用和最终回复 |
| 可测试 | 工作流节点、路由策略、Memory、RAG 和风险策略需要可独立测试 |
| 可扩展 | 新增设备平台、Agent 或工具时不修改已有 Agent 核心逻辑 |

## 20. 原型设计建议

后续 UI 原型可以增加以下视图：

- Agent 任务计划：展示任务目标、步骤、当前状态和耗时。
- 等待确认状态：展示操作、风险等级、影响范围和确认按钮。
- 任务恢复提示：应用重启后提示存在未完成任务，可继续或结束。
- 知识引用区域：最终回复中展示引用的文档和历史案例。
- 诊断报告页面：展示摘要、健康评分、证据、根因、建议和执行记录。
- Agent 轨迹详情：展示任务级思考说明、工具调用和验证结果，不展示模型私有推理。
- 存储与 Memory 设置：展示会话、任务、知识库的磁盘占用和清理策略。
- Agent 与模型设置：配置不同 Agent 使用的模型、工具范围、最大步骤和 Token 预算。

## 21. 需求边界

- 第一阶段不要求一次完成所有多 Agent 能力。
- 不允许为了 Agent 编排破坏现有设备、日志、文件和应用接口。
- 不将完整模型内部推理过程保存或展示给用户。
- 不允许 Agent 绕过已有敏感操作授权策略。
- 不要求任意模型支持完整 1M Token 单次输入；1M Token 是持久化和检索容量目标。
- 多 Agent 必须在单 Agent 固定工作流稳定后逐步引入。
- 所有新增能力应优先复用现有服务，通过适配器暴露为领域工具。
