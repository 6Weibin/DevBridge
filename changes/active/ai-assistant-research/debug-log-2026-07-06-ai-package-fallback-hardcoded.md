## 假设与验证

| 假设 | 验证结果 |
| --- | --- |
| 固定格式来自 Prompt | 否。固定表格来自 `toolFallbackAnswer` 的 Java 兜底逻辑，不是模型输出。 |
| 包名分类由 AI 完成 | 否。`packageType` 使用固定前缀判断，绕过了模型理解。 |
| 可以直接改成联网查询 | 当前工程没有互联网检索工具；直接把包名送外部查询会引入隐私、稳定性和授权边界问题，本次不新增。 |
| 删除固定分类会导致无回复 | 否。兜底仍展示工具原始输出；正常情况下 Provider 工具调用后继续输出时，仍由模型生成最终分析。 |

## 根因定位

普通对话流式响应在工具调用后使用 `contentAfterTool` 判断模型是否给出最终总结。部分 Provider 只执行工具但不继续输出，于是后端加入了兜底回复。原兜底把 `pm list packages` 输出识别出来后直接生成固定 Markdown 表格，并用包名前缀分类应用类型。这个设计把“业务理解和解释”放进了后端固定规则，导致 AI 能力表现被硬编码覆盖，也容易误判具体应用。

## 修复方案

移除 `packageNames`、`packageTable`、`packageType`、`packageBasis`、`escapeTable` 相关逻辑。`toolFallbackAnswer` 现在只展示最后一个工具结果的原始 stdout/stderr，并明确说明“模型未返回最终分析”。系统提示词补充包名分析约束，正常模型回复路径由 AI 基于工具输出自行解释，不确定时必须说明不确定。

## 验证结果

- `mvn -Dtest=AiConversationServiceTest test` 通过。
- `rg` 确认 AI 对话服务中不再存在固定包名表格和前缀分类逻辑。
