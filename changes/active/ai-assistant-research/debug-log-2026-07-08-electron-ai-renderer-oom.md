## 假设与验证

| 假设 | 置信度 | 验证结果 |
| --- | --- | --- |
| 单条 AI 回复和 SSE 下发上限仍过高 | 高 | 前端和服务端都允许 240k 级别文本，流式阶段会多次复制字符串并触发 React 渲染。 |
| Markdown 实时解析在长回复中生成过多临时对象和 DOM 节点 | 高 | 解析上限原为 200k，长表格/列表会生成大量数组和 React 节点，Electron 下容易放大内存。 |

| 单纯后端 Provider 输出异常导致 OOM | 低 | 前后端已有传输上限，崩溃表现更符合渲染层反复解析和驻留对象累积。 |

## 根因链路

Electron 渲染进程的 V8 堆达到 4GB，说明问题发生在前端内存，而不是 Java 后端。AI 助手在长回复、工具输出或大表格场景中，会经历 SSE 增量接收、字符串拼接、React 状态更新、Markdown 解析、表格/列表 DOM 生成等多个内存放大步骤。原有保护主要限制了传输和单条消息长度，但 Markdown 实时解析和 DOM 生成仍可能在长内容下产生大量临时对象，最终导致 Electron OOM。

## 修复方案

- 将前端 SSE 接收上限从 240k 降到 120k。
- 将服务端普通对话流式下发上限从 240k 降到 120k。
- 将前端单条 AI 消息驻留上限从 300k 降到 120k。
- 将工具输出、过程详情、确认续跑输出和历史持久化上限同步收窄。
- Markdown 渲染增加流式解析上限，流式阶段超过 36k 后切换轻量文本模式。
- Markdown 最终解析上限从 200k 降到 90k，显示上限从 260k 降到 120k。
- 表格最多渲染 240 行，列表最多渲染 320 项，内容块最多渲染 700 个，避免模型输出大表格时生成海量 DOM。
