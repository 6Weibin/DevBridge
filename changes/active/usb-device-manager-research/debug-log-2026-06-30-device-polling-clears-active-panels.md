# 设备轮询影响日志与文件面板排查记录

## 问题现象

- 设备列表定时刷新 `/api/devices` 后，实时日志页面的日志数据会被影响。
- 用户期望定时刷新只更新设备列表、头部设备信息、电量和连接状态。
- 文件管理和实时日志不应被轮询刷新重置；只有设备断开时在当前页面弹出提示。

## 排查步骤

- 阅读 `DevBridge-Front/src/app/App.tsx` 中设备轮询逻辑。
- 检查 `applyDeviceSnapshot`、`refreshDeviceSnapshot`、`refresh`、`loadFiles`、`startLogStream` 的状态更新范围。
- 验证前端构建、后端测试和后端静态资源打包。

## 假设与验证

| 假设 | 验证结果 | 结论 |
| --- | --- | --- |
| `/api/devices` 后端接口清空日志 | 后端接口只返回设备列表 | 排除 |
| 轮询直接调用 `setLogs` | 轮询未直接调用，但完整刷新会设置演示日志 | 部分相关 |
| 轮询替换选中设备并清空文件上下文 | 原实现设备为空时会清空 `remoteFiles`，并用基础快照替换选中设备 | 确认 |
| 设备断开时应保留当前文件/日志面板 | 用户明确要求弹出提示框而不是重置面板 | 确认 |

## 根因链路

1. 定时任务调用 `/api/devices` 获取设备快照。
2. 原 `applyDeviceSnapshot` 对设备为空场景直接清空文件树，并把选中设备切到等待设备。
3. 选中设备被替换后，文件/日志页上下文容易被联动逻辑影响。
4. 后端在线但设备不 connected 时，日志启动逻辑还可能误进入演示流。
5. 这些行为超出了“设备连接状态同步”的职责边界。

## 修复方案

- 新增 `mergeDeviceSnapshot`，轮询只合并设备基础状态，保留详情、电量、文件和日志上下文。
- 设备断开时保留当前选中设备，状态改为 `offline`，设备列表中移除该设备。
- 文件管理和实时日志页在设备断开时保留内容，并弹出“设备连接已断开”提示。
- 后端在线但设备断开时不再启动演示日志流。
- 手动刷新在日志采集中不再重置日志数据。
