# 设备切换后页签未保留排查记录

## 排查步骤

- [x] 复核用户复现路径：设备 A 选择实时日志，设备 B 选择文件管理，切回设备 A 后页签回到设备信息。
- [x] 搜索 `App.tsx` 中 `setSel`、`setTab` 和设备列表点击逻辑。
- [x] 确认设备列表点击事件中存在 `setTab("info")`，会强制覆盖目标设备的历史页签。
- [x] 增加设备维度页签记忆，并替换页签点击、设备点击和离线弹窗返回入口。
- [x] 执行前端构建、后端打包、服务重启和 HTTP 验证。

## 假设与验证

### 假设 1：页签状态被设备详情轮询重置

验证结果：不成立。设备详情轮询只更新 `sel` 详情，不直接写 `tab`。

### 假设 2：设备列表点击时强制回到设备信息

验证结果：成立。设备列表按钮原逻辑为 `setSel(d); setTab("info");`，每次从列表切换设备都会丢失该设备上一次使用的页签。

### 假设 3：只删除 `setTab("info")` 即可

验证结果：不成立。删除后只能保留全局当前页签，无法满足“Android 记住实时日志、iOS 记住文件管理”的设备维度诉求。

## 根因定位链

症状：多设备间切换时，目标设备没有恢复该设备最后选择的操作页签。

直接原因：设备列表点击事件强制设置 `tab = info`。

根因：前端 `tab` 是全局状态，没有维护 `deviceKey -> tab` 的设备维度 UI 上下文。

## 修复方案

- 新增 `tabByDeviceRef`，按 `platform:serial` 保存每台设备最后选择的页签。
- 新增 `selectTab(nextTab)`，页签点击时同步写入当前设备的历史页签。
- 新增 `selectDevice(device)`，切换设备前保存当前设备页签，切换后恢复目标设备历史页签，默认回到 `info`。
- 将设备列表点击、页签点击和离线弹窗“返回设备信息”统一接入上述方法。

## 验证结果

- `DevBridge-Server` 执行 `mvn package -DskipTests` 通过。

## 限制

当前验证以代码路径和服务可用性为主；真实多设备来回点击体验需要在用户当前连接设备环境中确认。
